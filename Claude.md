# 살아있는 연대기 (가칭) — 프로젝트 스펙

에이전트가 며칠~몇 달에 걸쳐 가상 도시의 역사를 스스로 확장해나가는 자기 진화 세계 시뮬레이션.
운영자(잭슨)는 신처럼 가끔 개입해 재난·예언·방향 전환을 던진다.

> 한 문장 정의: **AI 소설 생성기가 아니라 역사학 시뮬레이터.**
> 감정 목표: **림월드** — 인물에게 애착이 쌓이고, 죽으면 안타깝고, 소원을 이루면 기쁜 경험.

---

## 1. 핵심 설계 원칙 (불변)

1. **진실과 기록의 분리.** 세계 엔진만 유일한 진실(`world/`)을 안다. 독자가 보는 모든 것은
   편향되고 제한된 "기록"(`chronicle/`, `records/`)이다. 기록끼리의 모순은 버그가 아니라 세계의 질감.
2. **진실은 append-only.** 기록은 얼마든지 틀리고 수정될 수 있지만, `world/`의 진실은 소급 수정 금지.
   정합성 검수의 기준점이기 때문. (신의 개입도 진실을 과거로 거슬러 조작할 수 없음)
3. **드라마 금지.** 에이전트의 목표는 "흥미로운 사건"이 아니라 "오늘 하루도 세계가 살아가게 하라".
   기본값은 시장 개장, 결혼, 가뭄, 새 우물 같은 작은 변화. 큰 사건은 텐션이 임계치를 넘을 때만.
4. **비가역성.** 인물은 죽으면 끝. 불탄 건물은 사라짐. 자원은 유한. 되돌리기 없음.
5. **주인공 없음.** 서사적 비중은 지정하지 않는다. 사건이 결정한다. 어부가 왕이 될 수도,
   왕이 역사책 한 줄이 될 수도 있다.
6. **빈칸의 원칙.** 역사가는 모르는 것을 아는 척하지 않는다. "원인은 기록되지 않았다"로 남긴다.
   사이클 실패 시 그날은 "기록이 소실된 날"로 처리 — 실패도 세계관으로 흡수.
7. **애착은 디테일과 시간에서 나온다.** 무용한 디테일(버릇, 좋아하는 음식), 일상의 반복 노출,
   흉터의 누적, 작은 소원의 성취/좌절이 애착 엔진. 초상화 없음 — 문장(紋章)과 텍스트만.

## 2. 실패 패턴 (설계가 막아야 하는 것)

- **유토피아 붕괴**: 갈등이 자꾸 원만하게 해소되며 세계가 밋밋해짐 → 텐션 시스템으로 방지
- **인플레이션**: 매번 더 큰 사건을 만들다 도시가 반복 멸망 → 드라마 금지 원칙 + 사건 규모 예산
- **모순**: 죽은 인물 부활, 지명 변경 → 구조화된 상태 파일 + 역사가 검수로 방지
  (단, "기록"의 모순은 허용 — 진실의 모순만 금지)

## 3. 아키텍처

**git 저장소 = 세계 그 자체. 웹사이트 = 그 세계의 렌더링.**

```
/
├── world/            # 진실. 세계 엔진만 쓰고 읽음. 사이트에 절대 노출 안 됨.
│   ├── characters.json
│   ├── places.json
│   ├── factions.json
│   ├── timeline.json     # append-only 사건 로그 (truth)
│   └── oracles.json      # 신탁 이력 (append-only)
├── chronicle/        # 공식 연대기. 역사가 에이전트 작성. 편향·오류 있을 수 있음.
│   └── cycle-NNN.md
├── records/          # 신문, 주민 편지·일기. 더 왜곡되고 단편적. (1단계부터)
├── engine/           # 프롬프트, 규칙서, 사이클 스크립트
│   ├── rules.md          # 규칙서 (§6)
│   ├── world-engine.md   # 세계 엔진 프롬프트
│   └── historian.md      # 역사가 프롬프트
└── site/             # Astro 정적 사이트
```

- 배포: GitHub Actions — 커밋 시 Astro 빌드 → 정적 호스팅. 서버·DB 없음.
- 사이클: 상태 읽기(압축) → 사건 생성 → 상태 diff + 산문 커밋 → 역사가 검수 → 사이트 빌드
- 검수 실패 시 재생성 1회, 재실패 시 "기록 소실" 처리
- 시간: 1사이클 = 세계 시간 7일

## 4. 에이전트 구성

| 에이전트 | 역할 | 도입 시점 |
|---|---|---|
| 세계 엔진 | 진실 갱신. 텐션 기반 사건 생성. 유일하게 `world/`를 씀 | 0단계 |
| 역사가 | 공식 연대기 작성 + 정합성 검수 + 주기적 시대 요약(컨텍스트 압축) | 0단계 |
| 신문사 | 그날의 뉴스. 속보와 루머 혼재. 1면=역사, **뒷면=일상**(애착 지면) | 1단계 |
| 주민 | 편지·일기·소문. 매우 제한된 정보만 앎 | 1단계 |

## 5. 인물 스키마 초안

```json
{
  "id": "rohan-the-smith",
  "name": "로한",
  "role": "대장장이",
  "district": "port-district",
  "status": "alive",
  "desires": {
    "surface": "대장간을 아들에게 물려주는 것",
    "hidden": "이 도시를 떠나고 싶다"
  },
  "secret": "10년 전 화재의 목격자",
  "wish": { "text": "죽은 아내의 고향 바다를 한 번 보는 것", "status": "open" },
  "quirks": [
    "비 오는 날마다 아내가 심은 사과나무 아래서 점심을 먹는다",
    "왼손잡이지만 사람들 앞에서는 오른손을 쓴다"
  ],
  "scars": [],
  "relationships": {
    "mira-the-baker": { "type": "채무", "note": "빵집 재건 자금을 빌렸다", "tension": 40 }
  },
  "tension": 25
}
```

- `district`: 거주 구역 (places.json의 구역 id). `origin_district`(선택): 출신 구역 — 거주 구역과 다를 때만.
- `relationships`: `type`은 표준 어휘(가족|채무|거래|우정|연정|원한|갈등|은혜|비밀공유)만,
  `note`는 자유 서술. 상세는 `engine/rules.md` 규칙 13.
- `quirks`: 서사 기능이 없는 무용한 디테일 2~3개. 애착의 원료. **필수.**
- `wish`: 추적 가능한 작은 소원. `{ text, status, note }` 구조, status는 open|fulfilled|lost.
  이력은 비가역(삭제 금지), open은 인물당 1개. 몇 달에 걸쳐 성취되거나 좌절됨. 이완의 리듬 담당.
- `scars`: 사건이 인물을 통과할 때마다 각인. 죽지 않아도 흔적이 남는다. `event_id`는 timeline 사건 id 참조.
  형식: `{ "cycle": 12, "event_id": "east-fire", "mark": "왼팔 화상, 불을 무서워하게 됨" }`
- `tension`: 사건 생성의 분기 기준. 인물·구역·세력 모두 보유.
- 인물 수: 시드 5명 시작, 실존 인물 상한 30명 + 배경 인구(수치로만 존재).

사건 스키마는 `truth` / `public_knowledge` 필드를 분리해둘 것 (발굴·루머 기능의 기반).

## 6. 규칙서

규칙서는 [engine/rules.md](engine/rules.md) 참조. (전문은 그곳에만 두고 여기 중복하지 않는다)

## 7. 웹사이트 구성 (Astro 정적)

| 페이지 | 내용 |
|---|---|
| 오늘의 소식 | 신문 렌더링. 1면 헤드라인 + **뒷면 일상 기사** + 부고란 + 날씨(신의 개입 전조) |
| 지도 | SVG 인터랙티브. 건물 생성·소실이 diff로 반영 |
| 인물 도감 | 카드형. 생존/사망, 관계 그래프, scars 누적 = 인생 지층. 등장 챕터 링크 |
| 연대기 아카이브 | 전체 타임라인. 역사가의 시대 요약이 목차 |
| 신탁의 기록 | 신(운영자) 개입 이력. 나중에 독자 투표 자리 |

**부고란이 이 프로젝트의 심장.** 죽은 인물을 알던 다른 주민이 쓴, 편향되고 애정 묻은 부고.

## 8. 신의 개입 (2단계)

텔레그램 → Claude Code 판단 엔진 (기존 이슈 파이프라인 패턴 재활용).
명령 유형: **재난 / 은총 / 예언(자기실현적 — 기록으로 퍼져 주민이 믿고 움직임) / 침묵**.
"모순" 개입은 기록 조작으로만 허용, 진실 조작 금지.
외부 데이터 주입 옵션: 실제 날씨·뉴스 헤드라인을 은유로 변환해 전조로 흘림.

## 9. 로드맵

- **0단계 (지금)**: 저장소 스키마 + 규칙서 v0 + 세계 엔진·역사가 프롬프트.
  자동화 없이 Claude Code로 **수동 5사이클** 실행 → 실패 패턴 관찰 → 규칙서 보강.
  ⚠️ 자동화 유혹 금지. 규칙서가 여물기 전 자동화하면 매일 망가진 세계를 보게 됨.
- **1단계**: GitHub Actions 크론 자동화 (매일 1사이클) + 신문·주민 에이전트 + Astro 사이트
- **2단계**: 텔레그램 신의 인터페이스
- **3단계**: 사이트 공개 + 주간 다이제스트 + 다음 개입 독자 투표 (서버리스 함수 1개)
- 뒤로 미룬 것: 발굴 시스템(스키마에 자리만 있음), 텐션 지도 시각화, 만약 시뮬레이션, 종말/환생

## 10. 0단계 첫 작업 순서

1. 저장소 초기화 + 위 디렉터리 구조 생성
2. `world/` 스키마 확정 (인물 → 사건 → 장소·세력 순)
3. 시드 데이터: 도시 이름, 구역 3개, 시드 주민 5명(스키마 검증 겸), 창시 신화 1개
4. `engine/rules.md` v0 작성
5. 세계 엔진·역사가 프롬프트 작성
6. 수동 사이클 1회 실행 → 산출물 보고 스키마·규칙 조정 → 5회 반복
