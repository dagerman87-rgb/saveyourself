# WO-010: 어항 전환 (1) — 실시간 시간 모델 + 이벤트 러너

## 배경

운영 방향 전환 확정: 주간 배치 세계 → 실시간 어항 마을. 미르한과 6주 역사·인물·규칙 체계는
전부 계승한다. 이 WO는 심장 이식이다 — 화면(뷰어)은 WO-011, 이번에는 시간 모델과 이벤트
러너를 만들어 화면 없이 로그만으로 세계가 실시간으로 뛰는 것을 검증한다.

이 WO는 WO-009를 대체한다. WO-009에서 구현한 claude-cli 백엔드·중복 실행 가드는 계승하고,
cycle.yml·digest.yml 미등록 버그 수정도 이 WO에 흡수한다. 실행 환경: 당분간 집컴 로컬
(회사컴 이전은 검증 후 별도).

## 확정된 설계 결정 (기획)

- 시간 1:1: 마을 시간 = 현실 시간. 마을의 밤은 현실의 밤이다.
- 기록력 연속: 기록력 1년 7주 첫날부터 실시간 전환. 주(週) 번호 = 기존 cycle 번호 연속
  (1사이클=7일 정의 유지 — 주간 배치의 단위로 존속)
- 이벤트 주기: 06:00~24:00 사이 랜덤 1~3시간 간격 (하루 약 8~14회). 00:00~06:00은 마을도
  잔다 — 이벤트 없음 (드물게 "밤 사건" 허용: 주 1회 이하)
- truth 정책 변경: 운영자 뷰(로컬 뷰어·로그)는 truth 전면 공개. 공개 사이트(mirhan-site)는
  기존 격리 유지. 창이 두 개다 — 신의 창과 독자의 창.
- 실존 인물 상한 30 → 10 (현재 6명. 죽거나 떠나면 자리가 생기고, 새 인물이 온다)

## 작업 범위

### 1. 데이터 구조 — 로그 이중화

- `world/log.jsonl` 신설 (append-only): 모든 이벤트가 실린다. 항목 스키마:

```json
{ "id": "evt-...", "ts": "ISO8601", "week": 7, "grade": 0,
  "log_line": "로한이 미라의 빚 일부를 갚았다",
  "truth": "...", "public_knowledge": "...",
  "dialogue": [ { "speaker": "rohan-the-smith", "line": "..." } ],
  "staging": [ { "actor": "rohan-the-smith", "move_to": "bakery", "action": "talk" } ],
  "affected": [], "state_diff_summary": "..." }
```

- `log_line`: 못 본 시간을 따라잡는 한 줄 — "누가 무엇을 했고 결과는" 형식. 뷰어 피드와 로그의 원천
- `dialogue`: 이벤트에 대화가 있으면 NPC 발화 스크립트 (뷰어가 말풍선으로 재생)
- `staging`: 연출 힌트 — 누가 어디로 가서 무엇을 하는지 (뷰어가 이동·연기로 재생)
- `timeline.json`: 등급 1+ 사건만 승격 기록 (기존 스키마 그대로 — 주간 연대기·신문·기존
  사이트가 읽는 원천이므로 무변경). 등급 0은 log.jsonl에만 남는다
- characters·axes·factions 등 상태 파일: 기존 그대로, 이벤트마다 갱신

### 2. 규칙서 개정 (엔진 대상 — 문구는 취지 유지 하에 정리 재량, 완료 보고에 전문)

- 시간 조항: 위 확정 설계(1:1, 주 연속, 이벤트 주기·야간 규정) 반영
- 사건 예산의 재조정: 이벤트 대부분은 등급 0(생활)이다. 등급 1+는 하루 최대 1건, 도시급(2)은
  주 1건 이하, 시대급(3)은 신탁 없이는 불가. 텐션·방출 규칙(4-1)의 "사이클" 단위는 "주" 단위로 읽는다
- 인물 상한 10 반영 (규칙 11)
- log_line 문체 조항: 과거형 한 줄, 사실만, 결과 포함 ("~하였으나 ~함" 허용)
- 대화 조항: dialogue는 인물의 quirks·관계·현재 텐션과 정합해야 한다. 진실을 아는 자만
  진실을 말할 수 있다 (truth 정보의 화자 제한)

### 3. 이벤트 러너 — `engine/event-runner.mjs`

- 상주 프로세스 (집컴에서 `node engine/event-runner.mjs`로 기동, 콘솔에 로그 출력):
  1. 다음 이벤트 시각 랜덤 결정 → 대기
  2. `claude -p` (Opus) 호출 — 엔진 프롬프트의 이벤트판: world/ 전체 + 최근 log 20건 +
     rules.md 입력, 위 스키마의 이벤트 1건 출력
  3. 기계 검증: append-only(기존 log·timeline 불변), 참조 무결성, canary 등록, 등급 예산(§2)
     위반 시 재생성 1회 → 재실패 시 해당 이벤트 폐기(세계 무변화, 다음 대기)
  4. log.jsonl append + (등급 1+면 timeline 승격) + 상태 갱신 + 커밋
- 역사가의 이벤트 단위 검수는 없다 (주간 배치로 이동 — 실시간성 우선, 기계 검증이 1차
  방어선). 단 canary·append-only·예산 검증은 이벤트마다 수행
- 중복·동시 실행 가드 (WO-009 계승)

### 4. 주간 배치 — `engine/weekly.mjs`

- 매주 일요일 (러너가 시각 감지, 별도 스케줄러 불요): 역사가가 그 주 timeline(등급 1+)과
  log 표본을 읽고 주간 검수(모순·canary·규칙 위반 사후 감사 — 위반 발견 시 세계 정지 플래그 +
  보고) 후 연대기 집필 → 신문 발행 → 사이트 배포 트리거
- 기존 역사가·신문 프롬프트를 주간 단위 입력으로 소폭 수정 (문체·편향·격리 원칙 불변)
- 주간 QA 다이제스트 생성 (기존 build-digest 계승 — 로그 통계 추가: 이벤트 수, 등급 분포)

### 5. GitHub 워크플로 정리 (WO-009 계승분)

- cycle.yml·digest.yml 미등록 버그 수정 (원인 1줄 보고), cycle.yml은 비상 스위치화, digest는
  주간 배치로 흡수되면 제거 가능 — 판단 후 보고
- deploy.yml 무변경 (주간 배치의 push가 트리거)

### 6. 시운전

- 러너 기동 후 테스트 모드(간격 5~10분)로 이벤트 5건 생성 → 정상 모드 복귀.
- 5건의 log_line·dialogue·staging 전문과 상태 변화를 완료 보고에 첨부 (QA 검수 대상)
