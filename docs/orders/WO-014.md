# WO-014: 기다림과 애착

## 배경

"내일 왜 다시 접속하는가"에 대한 답. 셋을 넣는다 — 기다리는 사건(카운트다운), ★ 애착 등록, 계절. 핵심 설계 원칙: 기대는 확정되지만 결과는 창발이다. 결혼식이 예정되었다고 결혼이 보장되지는 않는다 — 카운트다운은 스포일러가 아니라 긴장 장치다.

## 작업 범위

### 1. 예정 사건 (카운트다운)

- `world/upcoming.json` 신설 (엔진 유지, append 후 상태 갱신):

```json
{ "id": "up-...", "title": "부두 보수 완공", "due": "ISO8601",
  "origin_event": "evt-...", "note": "조합이 약속한 기한",
  "status": "pending", "resolved_by": null }
```

- 규칙 신설 (엔진 대상, 취지 유지 재량 정리):
  - 예정 사건은 사건을 통해 태어난다 — 엔진이 이벤트 생성 시 자연스러운 기약(완공일, 혼례일, 출항일, 장날, 해산일)을 upcoming으로 등록할 수 있다. 강제 아님, 주 0~2건 권장
  - due 도래 시 러너가 그 예정을 최우선 입력으로 결과 이벤트를 생성한다. 결과는 창발: 성사·변형·연기·무산 모두 가능하며, 무산도 사건이다
  - 결과 이벤트는 upcoming을 resolved로 갱신하고 resolved_by에 event id 기록. pending인 채 due를 오래 지난 예정은 다음 이벤트에서 반드시 다룬다
- 뷰어: 사이드에 "다가오는 일" 카드 — 제목 + 남은 시간 카운트다운 (D-3, 6시간 뒤…). resolved되면 결과 log_line과 함께 완료 표시로 잠시 남았다 사라짐
- 첫 예정 후보를 심지 않는다 — 엔진이 스스로 첫 기약을 만들 때까지 기다린다 (기획이 심으면 창발 검증이 안 됨)

### 2. ★ 애착 등록

- 인물 카드에 ★ 토글 (localStorage). 효과:
  - 마을 로그에서 ★ 인물 관련 항목 하이라이트 (테두리 색)
  - ★ 인물의 새 이벤트 발생 시 브라우저 알림 (Notification API — 권한 요청 UI, 끌 수 있음). 알림 문구는 log_line 그대로
  - 지도에서 ★ 인물에 미세 표식 (이름표 옆 별)
- 엔진·세계 데이터 무변경 — 순수 뷰어 기능

### 3. 계절 정본화 (WO-012 제안 채택분)

- `world/clock.json`에 계절 매핑 추가 (한국 기준: 3~5 봄 / 6~8 여름 / 9~11 가을 / 12~2 겨울). 러너·뷰어·규칙이 이 정본을 읽는다 (러너의 월 하드코딩 제거)
- 규칙 조항 신설: "계절은 생업과 사건에 스며든다. 여름의 생선은 빨리 상하고, 겨울의 배는 묶인다. 계절 언급은 clock.json의 현재 계절과 정합해야 한다"
- 뷰어: 계절별 팔레트 미세 변화 (여름 짙은 초록 → 가을 마른 풀·낙엽 점묘 → 겨울 눈 쌓임·바다 회색 → 봄 꽃 점묘). 지금은 여름이므로 여름 기준 확인, 나머지는 강제 프리뷰 수단(콘솔 또는 쿼리 파라미터)으로 검증

## 범위 밖 (하지 말 것)

- 첫 예정 사건의 수동 등록 (엔진 창발 대기)
- 공개 사이트 변경 (upcoming은 신의 창 전용 — 사이트 노출은 관전 모드 때 결정)
- 노화 시스템 (보류 확정 사항)

## 검증 방법 (완료 보고에 포함)

1. 러너의 예정 처리 경로 설명 + (자연 발생 시) 첫 upcoming 항목
2. ★ 알림 실제 발화 1건 (다음 자연 이벤트로)
3. 4계절 프리뷰 스크린샷 또는 운영자 확인 지점
4. 모호 지점·다르게 처리한 부분·제안
